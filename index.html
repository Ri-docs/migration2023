RapidIdentity Migration Cheat Sheet

Pre-Migration Planning Live Execution Lessons Learned JP Script Method Validation Troubleshooting

0. Pre-Migration

  • Create new VMs for ARMS, DSS, and DB using the RI ISO. During install choose "Other versions" and select the exact production build. Assign temporary IPs to each new VM.
  • All old (VMware) and new (Nutanix) VMs must run the same RapidIdentity build.
  • Create snapshots of every old VM. These are your rollback points and protect against accidentally overwriting the running DB.
  • Gather admin credentials, DB passwords, and any config passwords stored in a vault.
  • Record current IPs, subnets (usually /24, RapidIdentity defaults to /8), hostnames, and gateways for every appliance.
  • Prepare an old-to-new server mapping list with hostnames and IPs for tracking.
  • Plan to apply the required DB prerequisite on the new DB appliance. This is required with the current ISO.

1. Planning

  1. Present the High-Level Flow

    Version parity -> DB prerequisite on new DB -> DB migration -> File-store migration -> Cutover (re-IP to production) -> Validation -> Upgrade to latest LTS.

  2. Version and Role Alignment

    Confirm builds and replicate appliance roles/capabilities from the old three nodes to the new three nodes.

  3. Network and Firewall Snapshot

    Document firewall rules (for example, port 5432/tcp for PostgreSQL) and verify new VMs allow required traffic.

  4. Hosts-File Strategy

    If the old DB VM uses /etc/hosts to define idautodb, replicate on the new DB VM pointing to itself.

2. Live Execution

2.1 Start the Database Service

  1. On the new DB VM, start the DB using the original password (from the old DB or config password). The Web UI will timeout until this is running.
  2. Connectivity test: nmap -p 5432 <DB-IP> from another appliance.

2.2 Apply the DB Prerequisite (Required)

  1. Enter Tech Mode on the new DB appliance and apply the prerequisite lines from the internal instructions provided with the ISO release notes.
  2. Restart services if instructed and confirm the DB is reachable on port 5432.

2.3 Replicate Capabilities and Restart RI

  1. Main Menu -> Status (1) -> Capabilities (5) on each old appliance; mirror selections on each new appliance.
  2. Restart services afterwards: 
    sudo systemctl restart ri.service
sudo systemctl status ri.service

2.4 Database Migration (UI-driven)

  1. Log in to Portal as idauto::admin / idautoAdmin.
  2. Create a snapshot of the old DB VM.
  3. Portal -> Config -> Systems -> Migrate Database -> Add Source. Point hostname to old DB.
  4. Get DB password from Status -> Advanced (7) -> rapidIdentity.properties -> Option 4.
  5. Select all modules except Audit; start migration.
  6. If you hit errors:
    • Network Mask Verification - check the subnet mask in Network Settings. If it is /8, confirm that is intended. For most on-prem networks use /24.
    • Schema mismatch - enable file-system export and run on the new DB: 
      psql idautodb idautoAdmin < /var/sftp/files/configdb.sql

2.5 File-Store Migration (CIFS)

  1. Old share path: Config -> General -> Settings -> File Storage.
  2. New share path: VM console Main Menu 6 -> Connection Info.
  3. Map both shares in Windows Explorer and copy everything from old -> new.
  4. In the new UI set File Storage to the new share.

2.6 Cut-Over (Re-IP or DNS)

  1. Preferred: re-IP the new appliances to the production IPs. Alternative: update DNS to point to the new app-node IPs.
  2. Shutdown order for old: ARMS -> DSS -> DB.
  3. Boot order for new: DB -> DSS -> ARMS/Portal.

2.7 Upgrade to Latest LTS

  1. After the cutover validates cleanly, upgrade the new servers to the latest LTS.
  2. Re-run the validation checks in section 5.

3. Lessons Learned (Real-Time Session)

  • RapidIdentity can display running even when DB is down. Always start the DB first.
  • Replicating appliance roles/capabilities early avoids Web UI 500 errors later.
  • Default /8 subnet masks break routing in most on-prem networks. Change to /24.
  • Firewall: new VMs may block 5432/tcp. Test with nmap or telnet <ip> 5432.
  • If versions misalign expect schema mismatches. Import configdb.sql then re-run migration.
  • Always verify new appliances point to the new CIFS share after copy.
  • Disable SSO temporarily if login loops occur after cutover.

4. Alternate Migration (JP Script)

JP's migrate-appliance.sh script can move data appliance-to-appliance without using the UI. The DB prerequisite on the new DB is still required with the current ISO.

  1. Download and Unpack

    cd /root
curl https://idauto-apps.s3.amazonaws.com/appliance-migrate/migrate-appliance.tar.gz | tar xvz

  2. Run on the New DB

    ./migrate-appliance.sh --src-host=<OLD-DB-IP> \
                       --target-db-host=localhost \
                       --migrate-db

  3. Run on Other Appliances

    ./migrate-appliance.sh --src-host=<OLD-ARMS-IP>

    Ensure /etc/hosts on each new appliance points idautodb to the new DB IP.

  4. Re-IP and Start Sequence

    1. Shut down old appliances (ARMS -> DSS -> DB).
    2. Assign their production IPs to the new appliances or update DNS to the new IPs.
    3. Boot order: DB first, then DSS, ARMS, Portal.

5. Validation

  1. Load Portal and Connect. Verify login and representative workflows.
  2. Monitor logs on all new nodes for errors.
  3. Confirm DNS or production IPs are correct and reachable. Retire old VMs.
  4. After upgrading to the latest LTS, re-run these checks.

6. Troubleshooting and Quick Commands

Common Issues

  • DB won't start - wrong password; pull from rapidIdentity.properties.
  • Portal 404/500 - DB down or hosts-file mis-pointed.
  • Schema mismatch - import configdb.sql then re-run migration.
  • Port 5432 blocked - open firewall or use nmap to verify.
  • SSO loops - disable SSO in properties, reboot services.

Cheat-Sheet Commands

# Restart RI
sudo systemctl restart ri.service

# Test DB port
nmap -p 5432 

# Export DB schema (old DB)
psql idautodb idautoAdmin > /var/sftp/files/configdb.sql

# Import schema (new DB)
psql idautodb idautoAdmin < /var/sftp/files/configdb.sql

# Find file if unknown location
find / -iname "configdb.sql" 2>/dev/null
